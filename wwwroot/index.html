Why it exists왜 태어났는가

.NET 10 opened the door. Cadenza walks five ways through it. .NET 10이 문을 열었다. Cadenza는 그 문을 다섯 갈래로 걸어 나간다.

dotnet run app.cs unlocked single-file execution — but raw. Cadenza shapes that primitive into a real workflow across console, worker, web, MCP and agent, so a single .cs becomes the whole program — and that program ships as one binary. Script to start, binary to finish. dotnet run app.cs는 "프로젝트 없이 단일 파일 실행"의 문을 열었지만 날것이었습니다. Cadenza는 그 원시 능력을 콘솔·워커·웹·MCP·에이전트 다섯 갈래에서 실제로 쓸 만하게 다듬어, 파일 하나가 곧 프로그램 전체가 되고 그 프로그램이 단일 바이너리로 배포되는 경험을 완성합니다. 스크립트로 시작해 바이너리로 끝납니다.

i.

Immediacy즉시성

No .csproj. No scaffolding. A single file runs as-is — as immediate as node app.js. .csproj도 스캐폴딩도 없이 파일 하나가 그대로 실행됩니다 — node app.js만큼 즉시.

ii.

Continuity연속성

Change one #:sdk line and the same model scales from shell scripts to workers, web APIs, MCP servers and local AI agents. #:sdk 한 줄만 바꾸면 셸 스크립트에서 워커·웹 API·MCP 서버·로컬 AI 에이전트까지 같은 모델로 확장됩니다.

iii.

Deployment배포

Self-contained (~30–40 MB) or AOT (~10–30 MB) single binaries — or publish the same file as an OCI image without a Dockerfile. 자족형(약 30–40MB) 또는 AOT(약 10–30MB) 단일 바이너리로 — 같은 파일을 Dockerfile 없이 OCI 이미지로 게시할 수도 있습니다.

Five SDKs, one model다섯 갈래, 하나의 모델

One #:sdk line decides what your file becomes. #:sdk 한 줄이 파일의 정체를 결정한다.

The same Tier-1 bare names (Glob, ReadText, Get, Tool, Log, Run) flow through every SDK. Pick the one that matches the shape of your program — the file stays one file. 같은 Tier-1 베어 네임(Glob, ReadText, Get, Tool, Log, Run)이 모든 SDK에 흐릅니다. 프로그램의 모양에 맞는 것을 고르세요 — 파일은 여전히 한 개입니다.

#:sdk Cadenza@1.0.15

Shell automation. CLI tools. One-off scripts. 셸 자동화. CLI 도구. 일회성 스크립트.

Replace bash + jq + awk with a single typed .cs. Bare names like Glob, ReadText, WriteLine work without imports. bash + jq + awk를 단일 타입 안전 .cs 한 장으로 대체하세요. Glob, ReadText, WriteLine 같은 베어 네임이 import 없이 동작합니다.

 count-files.cs
#!/usr/bin/env dotnet run
#:sdk Cadenza@1.0.15

foreach (var file in Glob("**/*.md"))
{
    var content = ReadText(file);
    WriteLine($"{file}: {content.Length:N0} bytes");
}
#:sdk Cadenza.Worker@1.0.15

Background services that respect Ctrl+C. Ctrl+C를 존중하는 백그라운드 서비스.

Run hosts a BackgroundService wired to ILogger. SIGTERM cancels the token. Just write the loop. Run이 BackgroundService를 호스팅하고 ILogger에 자동 배선됩니다. SIGTERM이 토큰을 취소합니다. 루프만 쓰세요.

 heartbeat.cs
#!/usr/bin/env dotnet run
#:sdk Cadenza.Worker@1.0.15

await Run(async (ct) =>
{
    while (!ct.IsCancellationRequested)
    {
        Log.Info($"Heartbeat at {DateTime.UtcNow:O}");
        await Task.Delay(TimeSpan.FromSeconds(30), ct);
    }
});
#:sdk Cadenza.Web@1.0.15

Minimal APIs without ceremony. 형식 절차 없는 Minimal API.

Get, Post, Put, Delete at top level. Records bind to JSON. Run starts Kestrel. 최상위 레벨의 Get, Post, Put, Delete. record가 JSON에 자동 바인딩되고, Run이 Kestrel을 띄웁니다.

Zero-install run무설치 실행

If they have the .NET SDK, they have your program. 받는 사람이 .NET SDK만 있다면 곧 당신의 프로그램을 가진 셈입니다.

Three steps from a fresh shell to a running service. No template, no project, no scaffolding. 새 셸에서 구동 중인 서비스까지 세 단계. 템플릿도, 프로젝트도, 스캐폴딩도 없습니다.

1

Create a file파일을 만든다

One #:sdk directive picks the runtime. Save as app.cs. #:sdk 한 줄로 런타임을 고릅니다. app.cs로 저장하세요.

2

Run it실행한다

Just dotnet run app.cs. The first run restores the SDK; subsequent runs start in < 1s. dotnet run app.cs만 입력. 첫 실행은 SDK를 복원하고, 이후는 1초 미만으로 부팅됩니다.

3

Ship it배포한다

Publish as a self-contained binary, or straight into a container image. Same file. Same command. 자족형 단일 바이너리로, 혹은 곧장 컨테이너 이미지로 게시합니다. 같은 파일, 같은 명령으로.

# 1. write $ cat > app.cs <<'EOF' #:sdk Cadenza.Web@1.0.15 Get("/", () => "Hello, Cadenza"); await Run(); EOF # 2. run $ dotnet run app.cs Now listening on: http://localhost:5000 # 3. ship (single binary) $ dotnet publish app.cs -c Release \ --self-contained -r linux-x64 → app (~32 MB, no .NET runtime needed)
The solo, taken to a container솔로 연주가 컨테이너째로

No orchestra — not even a Dockerfile. 오케스트라도, Dockerfile조차도 없이.

One file plays it all the way to a container. Built into the .NET SDK — Cadenza just lets you point it at a .cs. 한 개의 파일이 컨테이너까지 혼자 연주합니다. .NET SDK에 내장된 기능을 .cs 하나에 그대로 겨눕니다.

$ dotnet publish app.cs --os linux --arch x64 /t:PublishContainer → registry.local/app:1.0 (built, tagged, pushed) $ docker run -p 8080:8080 registry.local/app:1.0 → Now listening on: http://0.0.0.0:8080

Bind to 0.0.0.0 inside the container so external requests can reach it. For the smallest images, combine AOT publish with an Alpine or chiseled base. 컨테이너 안의 서버는 0.0.0.0에 바인딩해야 외부에서 닿을 수 있습니다. 가장 작은 이미지를 원한다면 AOT 게시와 Alpine 또는 chiseled 베이스를 조합하세요.

How it's different무엇이 다른가

Built on the platform, not beside it. 플랫폼 위에 올라탔지, 옆에 따로 서 있지 않다.

Native, not hosted네이티브, 별도 호스트 아님 vs dotnet-script / CS-Script

Cadenza sits on .NET 10's native file-based apps — no custom runtime, no separate host. It evolves with the platform and needs no extra install. Cadenza는 .NET 10의 네이티브 file-based app 위에 있습니다 — 별도 런타임도, 호스트도 없습니다. 플랫폼과 함께 진화하고 추가 설치가 필요 없습니다.

Real IntelliSense, no Dev Kit needed풀 IntelliSense, Dev Kit 없이도

Cadenza doesn't ship its own language layer — it rides the base Roslyn LSP, so completion in VS Code, Cursor, Kiro and friends just lights up. 자체 언어 레이어를 만들지 않고 베이스 Roslyn LSP에 올라타기 때문에, VS Code, Cursor, Kiro 등에서 완성이 자연스럽게 점등됩니다.

Exact-version SDKs정확 버전 SDK

#:sdk Foo@1.0.15 pins precisely — every machine sees the same SDK. This is reproducibility, not friction. #:sdk Foo@1.0.15는 정확히 고정합니다 — 모든 머신이 같은 SDK를 봅니다. 결함이 아니라 재현성입니다.

A growth path, not a dead end막다른 길이 아닌 성장 경로

When the script outgrows the file, dotnet project convert graduates it into a full project — like a cadenza written down from improvisation. 스크립트가 파일 한 장을 벗어날 때, dotnet project convert가 그것을 풀 프로젝트로 졸업시킵니다 — 즉흥에서 출발해 악보로 적힌 카덴차처럼.

Honest boundaries정직한 경계

When Cadenza isn't the right tool. Cadenza가 답이 아닐 때.

  • Multi-project solutions, library DLLs, or anything that needs a full .csproj — use the regular .NET project model. 다중 프로젝트 솔루션, DLL로 배포되는 라이브러리, 풀 .csproj가 필요한 경우엔 정규 .NET 프로젝트 모델을 쓰세요.
  • Cadenza is for the case where one file equals the whole program. Cadenza는 "단일 파일 = 프로그램 전체"인 경우를 위한 도구입니다.
  • Debugging inside a forked C# extension is restricted by the vsdbg license — for forks that need stepping, point at the netcoredbg path documented in the project. 포크 C# 확장의 디버깅은 vsdbg 라이선스 제약을 받습니다 — 스테핑이 필요한 포크는 프로젝트에 문서화된 netcoredbg 경로로 분기하세요.
